- **文档版本**：V1.0

## 1. 项目背景

大学生在求职季通常会同时申请多个岗位，需要在短时间内管理大量信息，包括：

- 公司与岗位信息
- 网申截止日期
- 简历、求职信、作品集等材料提交状态
- 笔试、面试、Offer 等阶段进展
- 与 HR 的沟通记录
- 不同岗位的优先级与跟进事项

当前常见做法是使用 Excel、备忘录、邮件、聊天记录等分散工具进行记录，导致以下问题：

1. **信息分散**：岗位链接、截止时间、材料版本、面试时间散落在不同地方。
2. **进度不透明**：无法快速判断每个申请当前处于什么阶段。
3. **易遗漏关键节点**：容易错过截止日期、面试、补材料等重要时间。
4. **多申请并行混乱**：同时申请多个岗位时，任务优先级不清晰。
5. **缺少复盘依据**：求职结束后，难以分析投递数量、转化率、渠道效果。

因此，需要设计一个 **“求职申请管理看板”平台**，以可视化、结构化的方式帮助用户完成从“发现岗位”到“拿到结果”的全流程管理。

## 2. 产品目标

### 2.1 产品愿景

构建一个面向大学生求职场景的轻量级申请管理平台，让用户可以在一个页面中清晰掌控所有申请进度，减少遗漏，提高执行效率。

### 2.2 核心目标

1. 统一管理所有求职申请信息。
2. 通过看板形式直观展示申请状态。
3. 通过提醒机制降低错过截止时间和面试的风险。
4. 支持记录材料、备注、跟进历史，形成完整申请档案。
5. 提供基础统计分析，支持用户复盘求职过程。

### 2.3 产品成功标准

- 用户可以在 3 分钟内完成第一条申请记录的创建。
- 用户可以在首页 10 秒内识别“最近最紧急的申请任务”。
- 用户可以在看板上直观看到所有申请所处阶段。
- 用户可以通过筛选和搜索在 5 秒内找到目标申请。

## 3. 目标用户

### 3.1 核心用户群体

1. **大学生实习求职者**  
   同时投递多个暑期实习、日常实习岗位。

2. **应届生校招求职者**  
   同时参与多个公司秋招/春招流程。

3. **有多轮流程管理需求的求职者**  
   需要管理笔试、一面、二面、HR 面、Offer 等复杂流程。

### 3.2 用户特征

- 申请岗位数量一般为 5~30 个，重度用户可达 50+。
- 使用招聘网站、企业官网、内推、校招群等多个渠道。
- 更关注效率、清晰性、提醒能力。
- 不希望平台复杂，偏好低学习成本、快速录入。

## 4. 用户痛点

### 4.1 信息管理痛点

- 不同公司信息分散在不同平台。
- 忘记自己投递的是哪一个版本的简历。
- 不确定哪些岗位还没有投递、哪些已经结束。

### 4.2 时间管理痛点

- 网申截止日期容易漏掉。
- 面试时间安排容易冲突。
- 没有统一的待办和提醒机制。

### 4.3 过程追踪痛点

- 无法快速知道某个岗位已经走到哪一步。
- 与 HR 的沟通内容没有地方统一记录。
- 同类岗位太多，难以区分优先级。

### 4.4 结果复盘痛点

- 不知道自己总共申请了多少岗位。
- 不知道哪个渠道的成功率更高。
- 不知道哪些岗位类型更匹配自己。

## 5. 产品定位

“求职申请管理看板”是一个 **面向大学生求职场景的 Web 端任务与流程管理平台**。

它不是通用项目管理工具，而是针对“求职申请”这个特定流程进行产品设计，重点解决以下问题：

- 申请信息记录
- 申请状态可视化
- 截止日期和面试提醒
- 材料与备注管理
- 求职过程复盘

## 6. 使用场景

### 场景 1：发现岗位后快速建档

用户在 Boss、LinkedIn、官网或内推渠道看到岗位，希望立刻记录：

- 公司名称
- 岗位名称
- 截止日期
- 岗位链接
- 当前状态（待投递）
- 优先级

### 场景 2：每日查看当日任务

用户打开平台，希望快速知道：

- 今天有哪些岗位快截止
- 哪些岗位还没投递
- 哪些岗位有即将开始的面试
- 哪些岗位需要继续跟进

### 场景 3：推进单个岗位申请

用户打开某个申请详情，希望查看或维护：

- 已提交哪些材料
- 当前阶段
- 面试安排
- 历史记录
- 备注与准备事项

### 场景 4：阶段流转管理

用户收到笔试/面试通知后，希望直接把申请从“已投递”拖到“笔试中”或“面试中”，系统自动保存状态变更时间。

### 场景 5：求职复盘

用户在一段时间后，希望统计：

- 总申请数
- 各阶段数量
- Offer 数
- 不同渠道带来的面试率/Offer 率

## 8. 核心业务流程

### 8.1 主流程

```text
发现岗位
→ 新建申请记录
→ 补充岗位信息与截止日期
→ 提交材料
→ 状态变更为“已投递”
→ 收到笔试通知
→ 状态变更为“笔试中”
→ 收到面试通知
→ 状态变更为“面试中”
→ 获得 Offer / 未通过 / 主动终止
→ 归档复盘
```

### 8.2 看板状态流转

建议默认状态如下：

1. **待收集**：感兴趣但还未整理完整信息
2. **待投递**：岗位信息已建立，但材料还未提交
3. **已投递**：已完成申请提交
4. **笔试中**：已进入笔试阶段
5. **面试中**：至少已进入一轮面试
6. **Offer**：收到录用结果
7. **未通过**：收到拒信或流程结束
8. **已归档**：主动关闭或已完成复盘

说明：
- 其中 1~5 为进行中状态。
- 6~8 为终态或半终态。
- MVP 支持默认状态；后续版本可支持自定义阶段。

## 9. 功能需求

## 9.1 账户系统

### 功能描述

支持用户注册、登录、退出登录，保证用户申请数据隔离。

### 功能点

- 邮箱注册/登录
- 密码登录
- 忘记密码（可选，MVP 可留接口）
- 退出登录

### 验收标准

- 未登录用户不可访问申请数据。
- 登录后只能访问自己的申请记录。

---

## 9.2 首页仪表盘

### 功能描述

首页提供申请概览与待办提醒，帮助用户第一时间掌握整体情况。

### 展示内容

- 总申请数
- 待投递数
- 已投递数
- 面试中数
- Offer 数
- 即将截止数（如 3 天内）
- 今日面试数
- 最近更新申请列表

### 交互要求

- 点击指标卡片，可跳转到对应筛选结果页。
- 即将截止和即将面试模块要高亮显示。

### 验收标准

- 用户在首页可快速获取当前申请整体进度。
- 首页加载时间在合理范围内，首屏信息清晰。

---

## 9.3 求职申请看板

### 功能描述

以 Kanban 看板形式展示所有申请，每列对应一个状态。

### 功能点

- 按状态分列展示申请卡片
- 支持拖拽卡片到不同状态列
- 每列显示申请数
- 支持卡片快速查看关键信息
- 支持卡片点击进入详情页

### 卡片展示字段

- 公司名称
- 岗位名称
- 截止日期
- 优先级
- 当前状态
- 最近更新时间
- 是否临近截止（标签提示）

### 交互要求

- 拖拽后状态立即更新。
- 拖拽成功后记录状态更新时间和历史日志。
- 临近截止的卡片要在视觉上突出。

### 验收标准

- 用户可通过拖拽完成状态流转。
- 状态变化后刷新页面仍能正确显示。

---

## 9.4 申请列表页

### 功能描述

除看板视图外，提供表格/列表视图，方便大量申请的检索和排序。

### 功能点

- 列表展示所有申请
- 支持分页
- 支持多字段排序
- 支持批量筛选

### 列表字段

- 公司名称
- 岗位名称
- 状态
- 截止日期
- 投递日期
- 面试日期
- 来源渠道
- 优先级
- 更新时间

### 验收标准

- 用户可在列表模式下高效定位目标申请。

---

## 9.5 新增/编辑申请

### 功能描述

用户可以创建新的求职申请记录，并对已有记录进行编辑。

### 必填字段

- 公司名称
- 岗位名称
- 当前状态

### 推荐字段

- 工作地点
- 来源渠道
- 岗位链接
- 截止日期
- 投递日期
- 优先级
- 薪资范围（可选）
- 备注

### 材料字段

- 简历已提交（是/否）
- 求职信已提交（是/否）
- 作品集已提交（是/否）
- 成绩单已提交（是/否）
- 其他材料说明

### 交互要求

- 新建表单应尽量简洁。
- 高级字段可折叠。
- 支持保存后跳转到详情页。

### 验收标准

- 用户可以成功创建、修改、删除申请。
- 数据校验明确，避免无效输入。

---

## 9.6 申请详情页

### 功能描述

展示单个申请的全部信息，并支持维护备注、历史、面试安排等内容。

### 页面模块

1. **基础信息区**
   - 公司名称
   - 岗位名称
   - 状态
   - 岗位链接
   - 来源渠道
   - 优先级

2. **时间信息区**
   - 截止日期
   - 投递日期
   - 下一场面试时间
   - 最近更新时间

3. **材料状态区**
   - 简历
   - 求职信
   - 作品集
   - 成绩单

4. **备注区**
   - 面试准备内容
   - 公司信息记录
   - HR 沟通记录
   - 其他补充说明

5. **历史记录区**
   - 状态变化历史
   - 时间节点变更记录
   - 操作日志

### 验收标准

- 用户可通过详情页完成单个申请的完整维护。
- 历史记录可追溯，不因编辑丢失关键时间点。

---

## 9.7 提醒系统

### 功能描述

为关键节点提供系统提醒，降低用户遗漏风险。

### 提醒类型

1. 截止日期提醒
2. 面试提醒
3. 长时间未更新提醒
4. 待补材料提醒

### 默认提醒规则

- 截止日期前 7 天提醒一次
- 截止日期前 3 天提醒一次
- 截止日期当天提醒一次
- 面试前 1 天提醒一次
- 面试前 1 小时提醒一次
- 7 天未更新的进行中申请提示跟进

### MVP 实现方式

- 站内通知 / 首页提醒模块
- 任务高亮展示

### 后续扩展方式

- 邮件提醒
- 日历同步提醒
- 浏览器通知

### 验收标准

- 即将截止和即将面试的申请能够在首页被清晰提示。
- 过期申请自动显示“已过截止日期”标签。

---

## 9.8 搜索、筛选与排序

### 功能描述

帮助用户在申请数量变多后快速找到目标记录。

### 搜索项

- 公司名称
- 岗位名称
- 备注关键词

### 筛选项

- 状态
- 来源渠道
- 优先级
- 是否临近截止
- 是否有面试安排
- 材料是否齐全

### 排序项

- 按截止日期升序/降序
- 按更新时间升序/降序
- 按投递日期升序/降序
- 按优先级排序

### 验收标准

- 用户可组合多个条件进行筛选。
- 筛选结果与看板/列表状态一致。

---

## 9.9 统计分析页

### 功能描述

提供基础数据统计，帮助用户了解申请进展并进行复盘。

### 指标设计

1. 总申请数
2. 各状态申请数
3. Offer 数
4. 面试率
5. Offer 转化率
6. 来源渠道分布
7. 本周/本月申请趋势

### 图表建议

- 状态分布：饼图 / 柱状图
- 申请趋势：折线图
- 渠道分布：柱状图
- 转化漏斗：待投递 → 已投递 → 笔试 → 面试 → Offer

### 验收标准

- 用户能从统计页快速了解当前求职整体表现。
- 图表数据与实际申请数据保持一致。

## 10. 非功能需求

## 10.1 易用性

- 页面结构简单直观，适合非专业用户。
- 新建申请操作应尽量少步骤。
- 看板和列表切换逻辑清晰。

## 10.2 性能

- 支持用户管理至少 100 条申请数据且基本操作流畅。
- 页面加载与状态更新应有较好响应速度。

## 10.3 安全性

- 用户数据隔离。
- 登录态校验。
- 后端接口需要权限控制。

## 10.4 可维护性

- 前后端模块清晰分层。
- 状态枚举、提醒规则、筛选条件尽量可配置。

## 10.5 可扩展性

- 后续支持附件上传、AI 分析、第三方日历集成。

## 14. API 需求（供开发实现参考）

以下接口为建议，不要求在 PRD 中写到代码级别，但为了“可运行平台”的落地性，这里给出 MVP API 设计。

## 14.1 认证接口

- `POST /api/auth/register` 注册
- `POST /api/auth/login` 登录
- `POST /api/auth/logout` 退出
- `GET /api/auth/me` 获取当前用户

## 14.2 申请接口

- `GET /api/applications` 获取申请列表
- `POST /api/applications` 创建申请
- `GET /api/applications/:id` 获取申请详情
- `PUT /api/applications/:id` 更新申请
- `DELETE /api/applications/:id` 删除申请
- `PATCH /api/applications/:id/status` 更新申请状态

## 14.3 历史记录接口

- `GET /api/applications/:id/history` 获取申请历史
- `POST /api/applications/:id/history` 添加日志/备注

## 14.4 提醒接口

- `GET /api/reminders` 获取提醒列表
- `PATCH /api/reminders/:id/read` 标记已读

## 14.5 统计接口

- `GET /api/stats/overview` 获取概览统计
- `GET /api/stats/funnel` 获取转化漏斗
- `GET /api/stats/channels` 获取渠道统计
- `GET /api/stats/trends` 获取投递趋势

## 15. 权限与约束

### 15.1 权限规则

- 用户只能查看和操作自己的申请。
- 未登录用户不可访问业务接口。
- 删除申请后应同步处理关联提醒与历史记录逻辑。

### 15.2 数据校验规则

- 公司名称不能为空。
- 岗位名称不能为空。
- 状态不能为空。
- 截止日期、面试日期必须为合法日期格式。
- 岗位链接如存在，应符合 URL 格式。

## 16. 关键交互说明

## 16.1 新建申请

用户点击“新增申请”后：

1. 打开表单弹窗或独立页面
2. 填写公司、岗位、状态等基础信息
3. 可选补充截止日期、链接、材料状态等
4. 点击保存
5. 系统生成申请记录并进入对应状态列

## 16.2 拖拽更新状态

用户将卡片从“已投递”拖至“面试中”后：

1. 前端更新卡片位置
2. 后端更新 status 字段
3. 生成一条 ApplicationHistory
4. 更新 updated_at
5. 如有关联提醒规则，可重新计算提醒

## 16.3 截止日期高亮逻辑

- 距离截止 > 7 天：正常显示
- 距离截止 ≤ 7 天：黄色提示
- 距离截止 ≤ 3 天：橙色提示
- 已过截止：红色提示 + “已过期”标签

## 20. 风险与应对

### 风险 1：用户录入成本太高

**问题**：字段太多会降低首次使用意愿。  
**应对**：MVP 采用“基础字段 + 高级字段折叠”设计。

### 风险 2：提醒过多造成打扰

**问题**：过多提醒会让用户忽视提醒。  
**应对**：MVP 只做关键提醒；后续支持用户自定义。

### 风险 3：状态流程不完全适配所有公司

**问题**：不同公司流程差异较大。  
**应对**：MVP 先用通用状态；后续支持自定义阶段。

### 风险 4：统计意义有限

**问题**：早期用户数据少时，统计页价值不明显。  
**应对**：先保证基础概览有价值，不做过度复杂分析。

## 21. 总结

“求职申请管理看板”平台的核心价值在于：

1. **统一管理**：集中记录所有求职申请信息。  
2. **可视化进度**：用看板清晰展示申请状态。  
3. **降低遗漏**：通过提醒和高亮机制管理关键时间点。  
4. **支持复盘**：通过统计页面帮助用户分析求职过程。  

从实现角度看，该产品具备明确的页面结构、数据模型、状态流转、API 设计和 MVP 验收标准，因此可以直接作为一个 **可运行 Web 平台** 的产品需求文档，用于后续原型设计、UI 设计与前后端开发。
